### 1. Processes vs. Threads (Concurrency)

Understanding the difference between these two units of execution is crucial. Both represent an entity running within a program's address space but differ significantly in overhead and isolation.



| Feature | Process | Thread |

| :--- | :--- | :--- |

| **Definition** | The smallest unit of **isolation**. It has its own memory space (data segment, heap). | The smallest unit of **multiprogramming** within a process. Shares the parent's code and data but has its own stack/Register state. |

| **Memory** | Dedicated; one copy per process. Accessing another process requires system calls (`fork`). |Shared among threads in the same process (except stacks). Context switch is cheaper due to less memory copying. |

| **Communication** | Requires Inter-Process Communication (IPC): pipes, sockets, shared memory. | Very easy via direct memory access; no IPC needed for data sharing within the same process. |

| **CPU Switching Cost**| High (requires saving/restoring full page tables, kernel state). | Low (only stack registers and thread-specific CPU state need to be saved). |



**Key Takeaway:** A single instance of a program is typically executed as one or more processes; within those processes, it may execute multiple threads. OS examtip: **Process = Safety/Isolation**, **Thread = Efficiency**.



---



### 2. Context Switching

This occurs when the CPU stops executing current process/thread and starts another from where it left off to manage time-sharing fairness.



#### The Sequence of Operations:

1.  **Trigger:** Either a timer interrupt (time sliceexpired) or a voluntary signal (I/O wait).

2.  **Save State:** The OS saves all relevant state information into the **Process Control Block (PCB)** for the *current* process/thread. This includes Program Counter, Register values, and Stack pointer.

3.  **Load State:** The OS retrieves saved state fromthe PCB of the *next* process to be scheduled. It loads registers, PC, etc., back into memory/CPU.

4.  **Resume Execution:** Control returns to the nextprocess at its stored program counter address (or jumps to a trap vector if interrupted).



#### Key Concepts:

*   **PCB (Process Control Block):** The data structure in RAM that holds all information about a single running or ready process/state/thread. It acts as the OS's "ID card."

*   **Swapping:** Involves moving processes between secondary and primary storage; it is much slower than just context switching registers but used for memory management when physical RAM is full.



---



### 3. Deadlock

Deadlock occurs when a set of processes are blocked because each process is holding a resource that anotherprocess needs, creating a circular wait condition where **no progress can be made**. All involved processesmust remain busy waiting indefinitely (though the system does not necessarily halt).



#### The Four Necessary Conditions:

To prove or analyze for deadlock existence in an exam, you often need to identify these four conditions occurring simultaneously (**Coffman's Conditions**):



1.  **Mutual Exclusion:** At least one resource must be non-preemptible (only one process can use it at a time). E.g., Physical printer is busy while waiting foranother user.

2.  **Hold and Wait:** A process holding at least oneresource is trying to acquire additional resources currently held by others, rather than releasing its current ones immediately.

3.  **No Preemption:** Resources cannot be forcibly taken away from a process; they must be released voluntarily upon completion of the specific operation or request. E.g., The printer does not release while printing for you until it finishes with that task.

4.  **Circular Wait:** A set of processes $\{p_1, p_2...\}$ are waiting in such a sequence that each process is holding one resource required by some other process and waits to be the next to acquire another resources held by the latter in its list: $P_i \to P_j$.



#### Prevention Strategies (Exams often ask for these):

To avoid deadlock, you can break **at least one** of the four conditions above. Common strategies include enforcing a strict ordering on resource acquisition or prohibiting holding while waiting to ensure no circular wait exists at all times.

### Time Complexity Analysis

*   **Time Complexity**: $O(E + V \log V)$ or more commonly simplified as $O((E+V) \log V)$ (since in dense graphs where $E > V$, it's often dominated by edges, but generally expressed using both). This assumes the priority queue is used effectively.
    *   **Initialization**: Initializing distances takes $O(V)$. Pushing all nodes initially to the heap can take $O(V \log V)$ or even optimized via binary heaps for each insertion step which adds overhead if not careful (standard approach uses $V$ insertions).
    *   **Main Loop**: We process every vertex exactly once ($V$ times). Inside the loop, we iterate over all edges connected to that vertex. For each edge $(u, v)$ with weight $w$, we update distances and perform a heap push operation: $O(\log V)$. Total pushes occur at most $E$ times (or fewer if updates are skipped due to visited checks).
    *   **Total**: The cost is proportional to the number of vertices processed multiplied by the log factor plus the total weight traversed. Thus, $T(V, E) = O(E \log V + V)$ or simply dominated by heap operations: **$O((V+E)\log V)$**.

*   **Space Complexity**: $O(V)$
    *   We need storage for `dist_map` to hold distances ($O(V)$).
    *   The Priority Queue can store up to $2E$ elements theoretically if many updates happen before popping, but practically limited by $V$. Thus space complexity is proportional to the size of input graph.

import heapq

def dijkstra(graph, start):
    """
    Find all shortest paths from 'start' to other nodes in an undirected graph.
    
    Args:
        graph (dict): Adjacency list where keys are source nodes and values 
                      are lists of [target_node, weight] tuples.
        start (int/str): The starting node ID.
    
    Returns:
        dict: A dictionary mapping each reachable node to its shortest distance from 'start'.
              If a node is unreachable, it maps to infinity.
    """
    # Initialize distances with infinity and set the start node's distance to 0
    dist = {node: float('inf') for node in graph}
    dist[start] = 0
    
    # Priority queue stores tuples of (current_distance, current_node)
    pq = [(dist[start], start)]
    
    while pq:
        d, u = heapq.heappop(pq)
        
        # If we've already found a shorter path to this node before processing it, skip
        if d > dist[u]:
            continue
        
        for v, weight in graph.get(u, []):
            new_dist = d + weight
            
            # Relaxation step: update distance if a shorter path is found
            if new_dist < dist[v]:
                dist[v] = new_dist
                heapq.heappush(pq, (dist[v], v))
    
    return dist
